# ADR-019 · 솔로 친화적 운영

**Status**: Accepted. 현재 유효. 2026-04-24 기준 이 ADR 은 모든 다른 ADR 의 **상위 기준**. 구체 구현은 ADR-001 ~ ADR-021 에 분산.

## 결론부터

이 템플릿의 모든 운영 결정의 상위 기준은 **"솔로 한 사람이 감당 가능한가?"** 예요. 기술적 우수성, 업계 표준, 확장성 가능성 같은 통념적 기준보다 **단일 개발자의 시간 · 주의력 예산** 이 우선. 동시에 **비목표** 를 명시적으로 선언해 "우리가 뭘 안 하는지" 를 분명히 밝혀요 (HA 99.99% · 멀티 리전 · 분산 추적 등).

## 왜 이런 고민이 시작됐나?

업계의 기술 선택 기준은 주로 **기업 환경 · 팀 규모** 를 전제로 해요.

- "마이크로서비스는 팀 분리에 유리"
- "코드 생성 도구는 수십 명 생산성 개선"
- "OpenTelemetry · 분산 추적은 운영 관측성 필수"
- "99.99% 가용성은 프로덕션 표준"

이런 표준을 **솔로 인디 개발자** 가 그대로 적용하면 어떻게 될까요?

- 앱 10개 × 마이크로서비스 3개 = 30개 서비스 운영. 새벽 장애 30곳에서 호출.
- 코드 생성 도구의 빌드 cycle × 매일 수백 번 = 시간 낭비.
- 분산 추적 백엔드 설정 + 유지 = 주당 몇 시간.
- 99.99% = 월 4분 허용. 서버 1대짜리 인디 앱이 감당 불가.

압력들이 부딪혀요.

**압력 A — 업계 표준의 유혹**  
링크드인 · 블로그 · 컨퍼런스가 "최신 · 정석 · 모범 사례" 를 말해요. 따라가고 싶은 압박.

**압력 B — 실제 제약**  
솔로 개발자의 시간 · 주의력 · 예산은 **매우 제한적**. 프롤로그 제약 1~3.

**압력 C — "나중에 확장" 유혹**  
"지금은 솔로지만 나중에 팀 커지면..." 가정으로 확장 대비 설계. 하지만 **그 "나중" 은 대부분 안 옴** — 인디 앱 80% 는 MVP 단계에서 멈춤.

**압력 D — 완벽주의의 함정**  
"제대로 하려면 이것도, 저것도" 하다 보면 **출시 자체를 못 함**. 완벽주의가 가장 큰 적.

이 결정이 답해야 했던 물음이에요.

> **솔로 인디 운영자에게 "좋은 시스템" 의 기준은 업계 표준과 어떻게 달라지는가?**

## 고민했던 대안들

### Option 1 — 업계 표준 그대로 따르기

마이크로서비스 · DDD · Clean Architecture · 분산 추적 · 99.99% 가용성 목표.

- **장점**: 이력서에 쓰기 좋음. 팀 확장 시 이행 비용 낮음.
- **단점 1**: **실제로 감당 불가**. 솔로가 10개 앱 × 각각 complex stack 을 운영 못 함.
- **단점 2**: 출시 지연. 완벽 추구하다 첫 앱도 못 내놓음.
- **단점 3**: "나중에 쓸 수도 있는 확장성" 에 시간 소모 — 대부분 쓸 일 없음.
- **탈락 이유**: 솔로 맥락에 대한 오해.

### Option 2 — "대충 하자" 로 완전히 기본적만

아키텍처 고민 없이 모든 화면에 HTTP + setState + 전역 변수.

- **장점**: 극도의 단순. 빠른 MVP.
- **단점 1**: **두 번째 앱** 에서 무너짐. 복붙 기반이 유지 불가.
- **단점 2**: 앱 공장 전략 (여러 앱 빠른 출시) 아예 불가 — 공통 자산 없음.
- **탈락 이유**: 중장기 운영 파탄.

### Option 3 — "솔로 감당 가능성" 을 명시적 기준으로 채택 ★ (채택)

모든 아키텍처 · 도구 · 운영 결정에 **"솔로 한 사람 감당 가능?"** 을 질문. No 면 기각, Yes 면 채택.

- **압력 A 대응**: 업계 표준을 참고하되 **맥락에 맞게 선택**. 감당 가능한 것만.
- **압력 B 대응**: 실제 제약을 원칙으로 공식화.
- **압력 C 대응**: "나중에 확장 가능성" 을 과중 평가하지 않음. 지금 필요한 것에 집중.
- **압력 D 대응**: **비목표** 를 명시적으로 선언 → "이건 안 한다" 공언 → 완벽주의 차단.

## 결정

### 운영 결정의 판단 기준

**모든 결정에 이 질문들을 적용**:

1. **솔로가 이걸 매일 유지할 수 있는가?** — NO 면 기각
2. **출시 후 새벽 2시 장애 대응 가능한가?** — NO 면 기각
3. **앱 10개로 늘어나도 감당 가능한가?** — NO 면 재설계
4. **1시간 안에 설정 · 이해 가능한가?** — NO 면 단순화

### 비목표 (Non-Goals) 명시

이 템플릿이 **의도적으로 안 하는** 것:

| 비목표 | 왜 안 하나? |
|--------|----------|
| **99.99% HA** | 월 4분 장애 허용. 1인 운영 불가. 99.5% (월 4시간) 가 현실적 목표 |
| **멀티 리전 배포** | 인디 앱 트래픽 규모에 과함. 서울 리전 하나로 충분 |
| **무중단 배포** | 배포 시 30초 다운타임 허용. 사용자 몰리는 시간 피해 배포 |
| **분산 추적 (Jaeger · Zipkin)** | 단일 백엔드라 분산 트레이싱 불필요. Sentry breadcrumb 로 충분 |
| **완전 자동화 (IaC for everything)** | Terraform · Ansible 유지 비용 > 이득. 수동 셋업 + 문서로 충분 |
| **서비스 메시 (Istio · Linkerd)** | 서비스 수 적어 메시 필요 없음 |
| **블루-그린 · 카나리 배포** | 내부 테스터 단계 활용 → 일반 배포 |
| **Kubernetes** | 단일 Mac mini · Fly.io 등 간단한 호스팅 충분 |
| **팀 확장성 (브랜치 전략 · 리뷰 워크플로우 복잡화)** | 솔로라 git flow 단순 `main + feature` 로 충분 |
| **코드 생성 파이프라인 (freezed · json_serializable)** | 솔로 cycle 에 부담 (ADR-004 참조) |

### 채택한 것 vs 거부한 것

**채택 (ADR 들이 구현)**:
- ✅ 단일 JVM 모놀리스 (백엔드) / 단일 Flutter 앱 (프론트)
- ✅ 수동 cherry-pick 전파 (ADR-001)
- ✅ Debug 폴백 서비스 (ADR-006)
- ✅ 관리형 서비스 선호 (Sentry · PostHog · Firebase · Supabase)
- ✅ 최소 인터페이스 (BootStep 7줄, Kit 계약 55줄)
- ✅ 정책 제공 + 기본값 (캐시 정책 5개, 로딩 UX 4개)

**거부**:
- ❌ Clean Architecture 4계층 (ADR-002)
- ❌ BLoC · GetX 상태관리 (ADR-005)
- ❌ OpenAPI codegen (ADR-009)
- ❌ 서비스 로케이터 · 2번째 DI (ADR-005)
- ❌ 마이크로서비스 (spring-backend-template ADR-001)

### 관리형 서비스 선호

자체 운영 vs 관리형의 결정:

| 영역 | 관리형 선택 | 이유 |
|------|---------|------|
| 크래시 리포팅 | Sentry | 자체 Kibana · ELK 운영은 솔로 불가 |
| 사용자 분석 | PostHog · Mixpanel | 자체 데이터 파이프라인 구축 피함 |
| 푸시 알림 | Firebase FCM | APNs · GCM 직접 연동 피함 |
| 백엔드 DB | Supabase · Firebase | self-hosted Postgres 운영 피함 |
| 배포 | Play Internal · TestFlight | 자체 OTA 서버 피함 |
| 모니터링 | Grafana Cloud · Sentry · PostHog | 자체 Prometheus · Alertmanager 피함 |

### 시간 예산 원칙

- **신규 앱 출시 준비**: ≤ 1일 (rename + 환경변수 + 첫 배포)
- **주당 운영 부담**: ≤ 4시간 (크래시 확인 · 리뷰 대응 · 업데이트)
- **월간 장애 대응**: ≤ 1회 1시간
- **문서 읽기**: 새 개발자가 2시간 안에 첫 배포 가능

이 예산을 초과하면 **설계 결정을 재고**.

### 설계 선택 포인트

**포인트 1 — "비목표" 를 ADR 에 명시**  
"이거 안 한다" 를 당당히 적어둬요. 숨기면 나중에 "왜 안 했지?" 질문 매번 받아야 함. 명시된 비목표는 **완벽주의 유혹 차단**.

**포인트 2 — "나중에 확장" 가정의 위험성**  
"지금은 솔로지만 팀 되면 마이크로서비스 전환" 같은 가정으로 설계하면 **지금의 복잡도 = 비용**, **미래 확장 = 어쩌면 안 올 이득**. 비대칭. 확장은 **실제로 필요해졌을 때** 하기.

**포인트 3 — 관리형 서비스 비용 vs 운영 시간**  
Sentry Pro $26/월 vs 자체 ELK 운영 (주 4시간 × 월 $X). 솔로 시간은 **복제 불가능한 자원** 이라 관리형 비용이 거의 항상 이득.

**포인트 4 — 완벽보다 "충분히 좋음"**  
99.9% 가용성이면 월 43분 장애 허용. 인디 앱엔 충분. 99.99% 를 위해 들어가는 추가 복잡도 (멀티 리전 · 자동 failover) 는 과투자.

**포인트 5 — "지금 필요한 것" 집중**  
Auth 없는 로컬 앱은 `AuthKit` 불필요. 백엔드 없는 앱은 `BackendApiKit` 불필요. ADR-003 의 FeatureKit 이 이 원칙의 구조적 표현.

## 이 선택이 가져온 것

### 긍정적 결과

- **실제 출시 가능**: 완벽주의 함정 회피. 앱 공장 작동.
- **유지 비용 제한**: 주당 4시간 내외로 N 개 앱 운영.
- **결정 속도**: "솔로 감당 가능?" 질문으로 빠른 판단.
- **확장 유예**: 필요해지면 그때 전환. 지금은 단순.
- **비목표 명시의 안도감**: "이거 안 하니까 걱정 마" — 사용자 · 본인 모두에게.

### 부정적 결과

- **이력서 불리**: "마이크로서비스 · Kubernetes 경험 없음" — 취업 관점에선 약점.
- **확장 시 재작업 가능**: "나중에 팀 커지면" 정말 오면 모놀리스 → 마이크로서비스 전환 비용. 이건 **실제 필요할 때만** 감당.
- **업계 동향과 거리**: 최신 hype (Deno · Bun · Rust 마이그레이션 등) 을 일부러 피함. 트렌드 놓칠 수도.
- **"이게 진짜 최선?" 의심**: 단순하게 해놓고 "더 좋은 방법 있을텐데" 하는 심리. 비목표 명시로 완화.

## 교훈

### 교훈 1 — 업계 표준은 **조직 규모 가정** 이 전제

컨퍼런스 · 블로그의 "모범 사례" 는 대부분 **수십 명 팀 · 대규모 트래픽** 을 가정해요. 솔로 인디에겐 **맥락이 다름**. 표준 맹신은 위험.

**교훈**: 모든 기술 결정 뒤엔 **"어떤 환경을 가정하는가?"** 가 있어요. 그 가정이 내 환경과 일치하는지 확인.

### 교훈 2 — "나중에 확장" 가정은 **대부분 실현 안 됨**

인디 앱 80% 가 MVP 단계에서 멈춰요. 성공한 15% 도 대부분 소규모 · 서버 1대로 충분. 진짜 스케일 문제 생긴 5% 는 그때 가서 해결. **미리 대비는 과투자**.

**교훈**: YAGNI (You Aren't Gonna Need It) 는 **솔로 환경에서 특히 중요**. 지금 필요한 것만 만들기.

### 교훈 3 — 관리형 서비스는 **시간 vs 돈 비대칭** 의 결과

돈 ($26/월 Sentry) 은 **거의 0 에 가깝게 만들 수 있음**. 시간은 **복제 불가**. 돈으로 시간을 사는 건 항상 이득.

**교훈**: 솔로 운영에서 **$X/월의 관리형 서비스** vs **주 Y시간의 자체 운영** 비교 시, Y 가 조금만 되어도 관리형이 이득. "공짜 셀프 호스팅" 유혹 경계.

### 교훈 4 — "비목표" 를 적어두면 완벽주의가 가벼워져요

"99.99% 가용성 추구" 같은 걸 암묵적으로 생각하다 보면 무한 작업. **"99.5% 가 목표" 라고 명시** 하면 "여기까지 OK" 심리적 안정. 당당한 목표 하향 조정이 오히려 생산성.

**교훈**: 명시적 비목표는 **정신 건강 도구** 이기도 해요. "이건 안 한다" 선언이 완벽주의를 차단.
